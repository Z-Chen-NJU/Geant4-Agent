## English

### 1. Objective

`Geant4-Agent` is a controlled first-pass configuration agent for Geant4-oriented simulation setup. The immediate goal is not to generate a full production-ready Geant4 application, but to close the first-pass loop for five domains:

- geometry
- materials
- source
- physics
- output

The system accepts natural-language requests, normalizes them into structured proposals, validates them against deterministic rules, and commits them into an auditable configuration state. The language model does not write directly to the final config.

### 2. Core Technical Route

The current route is built around five principles:

1. LLM-first for semantic normalization.
2. Single source of truth for committed configuration state.
3. Arbiter + validator gate as the only commit path.
4. Operator-graph geometry instead of per-scenario geometry templates.
5. Human approval for destructive overwrite.

This is a control-plane architecture, not a pure prompting workflow.

### 3. Runtime Pipeline

A single user turn is processed as follows:

1. The UI sends the current user message and the existing session state to the orchestrator.
2. The LLM normalization layer extracts intent, slot frame, semantic targets, and candidate updates.
3. BERT-based components provide conservative structure priors and fallback cues. They do not own the final decision.
4. Runtime semantic synthesis builds geometry candidates, graph programs, and grounded parameter proposals.
5. The arbiter merges user-grounded proposals, rejects unsafe conflicts, and enforces overwrite confirmation.
6. The validator gate checks schema completeness, semantic consistency, and feasibility constraints.
7. Only gated results are committed into the canonical config.
8. The dialogue layer renders a user-facing response, using action-level templates plus controlled LLM naturalization.

### 4. Why the System Is Not LLM-Only

A pure LLM flow failed for the expected reasons in this problem class:

- implicit overwrites across turns
- geometry drift after partial clarification
- internal field leakage into user-visible replies
- hallucinated defaults that looked plausible but were not user-specified
- inconsistent bindings between geometry, materials, and source settings

The current design therefore restricts the LLM to proposal generation. Commit authority remains in deterministic components.

### 5. Role of Each Major Layer

#### 5.1 LLM layer

The LLM is responsible for semantic normalization, likely intent extraction, candidate field proposal, and limited naturalization of follow-up questions and summaries. It is not allowed to commit final configuration directly, override locked values without explicit user intent, or decide feasibility on its own.

#### 5.2 BERT layer

The BERT layer is intentionally downgraded from a central classifier to a supporting prior. It currently contributes structure prior, fallback semantic cues, and auxiliary confidence signals for routing.

#### 5.3 Orchestrator and arbiter

The orchestrator coordinates session state, pending overwrite flow, dialogue phase, and candidate lifecycle. The arbiter resolves candidate conflicts and ensures that only defensible updates survive to validation.

#### 5.4 Validator gate

The validator gate is the hard boundary before commit. It checks required-field completeness, path legality, semantic consistency, geometry feasibility rules, and cross-field coherence.

#### 5.5 Dialogue layer

The dialogue layer turns internal state into clarification questions, overwrite confirmation prompts, rejection acknowledgements, progress summaries, and final completion messages. Action templates are used here to keep outputs stable while preserving limited naturalization.

### 6. Geometry Route

Geometry is the most specialized part of the stack. The route has two levels.

#### 6.1 Single-solid families

Current single-solid families include:

- `single_box`
- `single_tubs`
- `single_sphere`
- `single_orb`
- `single_cons`
- `single_trd`
- `single_polycone`
- `single_cuttubs`
- `single_trap`
- `single_para`
- `single_torus`
- `single_ellipsoid`
- `single_elltube`
- `single_polyhedra`

#### 6.2 Operator-graph families

For composite layouts, the system uses graph-style assembly operators rather than per-scenario templates. Current graph families include:

- `ring`
- `grid`
- `nest`
- `stack`
- `shell`
- `boolean`

This route is aligned with the original project requirement: finite assembly operators plus constraints, instead of scene-specific template classes.

### 7. Why Operator-Graph Matters

A structure label alone does not scale well to real Geant4 requests. The operator-graph route generalizes better across scenarios with similar assembly semantics, and it supports partial completion and family-aware questioning without pretending the scene is already fully known.

### 8. Current Validation Strategy

The system currently uses conservative validation rather than full Geant4 execution:

- parameter legality checks
- containment checks
- spacing checks for grid and ring
- AABB-based sufficient conditions
- structure-family completeness checks
- overwrite and lock rules

This is intentional. The current agent is a first-pass validator, not yet the final Geant4 overlap judge.

### 9. Current Evidence

Current controlled evaluation tracks:

- `v3_24`: structured multi-turn live regression baseline
- `v4_24`: colloquial-material companion regression set

At the current checkpoint, both `v3_24` and `v4_24` have reached `24/24` in live regression. `v3_24` remains the main baseline, while `v4_24` is retained as the stricter colloquial-material companion set.

### 10. Known Remaining Gaps

The main remaining gaps are coverage gaps, not architecture gaps:

- broader colloquial material grounding beyond the current casebank
- more adversarial multi-turn stress cases
- further operator-graph coverage under fuzzy language
- larger public-facing regression bundles
- tighter internal-leak scoring that measures user-visible leakage only

### 11. Recommended Next Steps

1. Continue expanding the live casebank instead of rewriting the architecture.
2. Improve colloquial material grounding in the LLM-to-runtime handoff.
3. Keep BERT as a conservative prior unless a new gap clearly requires retraining.
4. Add more graph-heavy and adversarial demo cases before any open public trial.
